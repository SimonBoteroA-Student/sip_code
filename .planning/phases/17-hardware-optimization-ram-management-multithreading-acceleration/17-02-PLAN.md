## Goal

Integrate MemoryMonitor into all three build functions (labels, IRIC, features), add lifecycle `del`+`gc.collect()` to free completed data structures, and implement checkpoint-and-resume on abort. The build function signatures are already extended with `n_jobs`/`max_ram_gb` parameters by Plan 17-01 — this plan fills in the actual implementation.

## must_haves

- [ ] `build_labels()` — MemoryMonitor integration: check before each chunk, adaptive chunk sizing, abort on critical
- [ ] `build_iric()` — MemoryMonitor integration: same pattern as labels
- [ ] `build_features()` — MemoryMonitor integration activated (signature already has `max_ram_gb`)
- [ ] Checkpoint save on abort for all three build steps (`artifacts/{step}/_checkpoint.parquet`)
- [ ] Checkpoint resume on restart: load processed IDs, skip already-processed rows
- [ ] Lifecycle `del` + `gc.collect()` added for exhausted lookups/accumulators in all 3 build steps
- [ ] Abort message shows current RSS, budget, and suggests increasing `max_ram_gb`
- [ ] All existing tests pass — no behavioral change when `max_ram_gb=None`

<task id="17-02-01" title="Integrate RAM management into build_labels()" wave="1" depends_on="" files="src/sip_engine/shared/data/label_builder.py">

### MemoryMonitor integration
- If `max_ram_gb` is not None, create `MemoryMonitor(max_ram_gb)` (imported from `shared/memory.py`)
- Before each chunk dispatch in the contratos streaming loop, call `monitor.check()`:
  - `"ok"`: proceed with current chunk_size
  - `"warning"`: adjust chunk_size via `adaptive_chunk_size()`, run `gc.collect()`
  - `"critical"`: run aggressive `gc.collect()`, retry once; if still critical, save checkpoint and abort with descriptive message showing current RSS, budget, and suggestion to increase `max_ram_gb`

### Checkpoint support
- Checkpoint path: `artifacts/labels/_checkpoint.parquet`
- On start: if checkpoint exists and `force=False`, call `load_checkpoint()` to get processed IDs set; skip rows with `id_contrato` in set
- On successful completion: call `remove_checkpoint()`

### Lifecycle cleanup
After label assignment sets (M1, M2 contract sets) are consumed → `del` + `gc.collect()`:
- After `m1_contracts` and `m2_contracts` are used to assign columns
- After `boletines_set` is consumed in `_compute_m3_m4()`
- After `all_rows` list is converted to DataFrame → `del all_rows`

<task id="17-02-02" title="Integrate RAM management into build_iric()" wave="1" depends_on="" files="src/sip_engine/classifiers/iric/pipeline.py">

### MemoryMonitor integration
Same pattern as labels: create monitor if `max_ram_gb` provided, check before each chunk in the main contratos loop (lines 282+), adapt chunk_size dynamically, abort with checkpoint on critical.

### Dynamic chunk_size
Pass `chunk_size` to `load_contratos(chunk_size=current_chunk_size)` in the streaming loop. Adjust between iterations based on `adaptive_chunk_size()`.

**Recommended approach:** Keep the single generator but pass the initial chunk_size. When memory pressure triggers a halving, the new chunk_size applies to the NEXT call to `load_contratos()` if restarted (checkpoint-resume naturally handles this). Within a single run, the chunk_size from the generator is fixed — the adaptive response is: on warning, `gc.collect()` aggressively; on critical, checkpoint and abort. The restart with a smaller chunk_size happens on resume.

### Checkpoint support
- Checkpoint path: `artifacts/iric/_checkpoint.parquet`
- Same skip-based resume pattern as labels

### Lifecycle cleanup
After all lookups are consumed in the streaming pass:
- `del procesos_lookup, num_actividades_lookup, bid_stats_lookup` + `gc.collect()`
- After `all_rows` → DataFrame: `del all_rows` + `gc.collect()`

## Verification Criteria

1. **MemoryMonitor active:** When `max_ram_gb` is provided, all three build steps create and use a MemoryMonitor
2. **Adaptive response:** At 90% memory, chunk size reduces and GC runs; at 100%, checkpoint saves and step aborts
3. **Checkpoint round-trip:** Save → abort → restart → load checkpoint → resume from last processed row → complete successfully
4. **Lifecycle cleanup:** After each build step completes, large lookup dicts are explicitly freed via `del` + `gc.collect()`
5. **Abort message:** Shows current RSS (GB), budget (GB), and actionable suggestion to increase `max_ram_gb`
6. **No-op when disabled:** `max_ram_gb=None` produces identical behavior to pre-Phase 17
7. **All 22+ test files pass:** `pytest tests/ -x -q` exits 0
